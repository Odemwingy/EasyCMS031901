# AI 引导文件（固定模板）：初始化项目结构与文档

## 人类开发落地总体流程（给人类看的，AI 可忽略）

> 本节用于指导**人类**如何落地一个项目的完整闭环；AI 可以忽略本节内容（不作为模板生成约束）。

### 1）需求调研阶段
- 根据需求进行调研与澄清，识别边界与优先级。
- 检验可实现的技术方案，确认关键风险点与约束（性能、安全、部署、成本等）。
- 最终确定技术架构与技术栈，并沉淀到需求文档/架构说明中。

### 2）开发阶段（前后端协作）
- **前端**：
  - 开发前端时使用 MCP 与 `ui/` 对接，按高还原度实现设计稿。
  - 组件与布局遵循统一规范，避免页面间不一致。
- **后端**：
  - 后端生成/维护 API 文档（接口路径、权限点、请求/响应字段、错误码等）。
  - 与前端对接接口契约，确保联调一致性。

### 3）测试阶段
- 测试通过后再进入下一步交付。
- 测试发现问题：提交 issue，并推动前后端修复与回归（保证闭环）。

### 4）需求变更（版本化）
- 确定需求变更后：**保留原需求文档**。
- 生成新的需求文档版本（例如 `PRODUCT_REQUIREMENTS_v2.md` 或 `PRODUCT_REQUIREMENTS.md` + 版本章节/变更记录）。
- 让 AI 对比 v1/v2 差异，输出差异清单与影响分析（页面、接口、数据、权限、测试用例等）。

### 5）人类/AI 交替开发协作
- 人类本地开发为主。
- 人类休息时：让 OpenClaw 拉取最新代码，基于 `memory/`、`progress/`、`issue/`、`ui/` 等上下文继续开发与完善。

## 0. 目的（必须理解）
本文件是**本仓库的固定模板规范**：用于在新项目中**原样生成**一套可被 AI/开发者接续的目录结构与初始化文件。

关键要求：
- **初始化内容必须固定**：按本文件给出的“标准内容”逐字复制生成；**不得**由 AI 自行推理、自由发挥或改写。
- 若需要变更模板内容，应先更新本仓库模板文件，再同步更新本引导文件中的标准内容。

## 1. 初始化后必须存在的目录与文件（固定清单）
AI 读完本文件后，必须在仓库里生成以下结构（路径固定，内容固定；“标准内容”见第 2 节）：  

- `.gitignore`（忽略依赖目录与本地私密配置，内容固定见 2.0 节）
- `PRODUCT_REQUIREMENTS.md`
- `rule.md`
- `AI_PRD_GUIDE.md`（本文件）
- `changelog/`（迭代记录目录，格式模仿 `issue/`：每条迭代一个文件，记录迭代日期与迭代内容）
  - `changelog/README.md`
- `memory/`
  - `memory/README.md`（memory 触发规则与格式规范）
  - `memory/YYYY-MM-DD.md`（初始化当日的记忆文件；后续按触发条件新增）
- `progress/`
  - `progress/PHASES.md`（阶段定义，仅在需求/阶段变更时更新）
  - `progress/CURRENT_PROGRESS.md`（**主控总览**：当前阶段 + 各端状态速览 + 主要阻塞项）
  - `progress/阶段N/`（**拆解任务后按阶段创建**；N 为阶段名，例如 `阶段一/`、`阶段二/`）
    - `progress/阶段N/OVERVIEW.md`（**本阶段落地总览**：细致拆解本期所有任务，是开发的直接依据）
    - `progress/阶段N/frontend.md`（前端领域进度，本阶段任务置顶）
    - `progress/阶段N/backend.md`（后端领域进度，本阶段任务置顶）
    - `progress/阶段N/ui.md`（UI 设计领域进度，本阶段任务置顶）
    - `progress/阶段N/tests.md`（测试领域进度，本阶段任务置顶）
- `app/`
  - `app/README.md`（占位说明；后续所有代码/产物均在 app 下）
- `issue/`
  - `issue/README.md`（占位说明；issue 文件按 `rule.md` 约定）
- `ui/`
  - `ui/README.md`（占位说明；UI 设计数据放这里）
- `config/`（本地私密配置目录）
  - `config/.gitkeep`（空占位；用于确保目录存在）
- `tests/`
  - `tests/README.md`（占位说明；测试用例放这里）
- `docs/`
  - `docs/README.md`（文档归档说明）
  - `docs/stage-N/`（阶段文档归档目录；如 `stage-1/`、`stage-2/`）
- `technology/`
  - `technology/README.md`（目录说明与使用规则）
  - `technology/TECH_STACK.md`（技术选型主文档；**开发前必读**）

### 2.4.1 线上方案时：`issue/README.md` 与 `ui/README.md` 作为 skill
- 当 `issue/` 或 `ui/` 使用线上方案（如 MCP、Figma、在线系统）时：
  - `issue/README.md` 与 `ui/README.md` 必须写成可执行的 skill 指南，明确如何连接、如何读取、如何使用。
  - README 只写参数名与接入步骤，不写死个人令牌。
- 各开发者的令牌/密钥等私密信息统一放本地 `config/` 目录（例如 `config/issue.local.env`、`config/ui.local.env`），不写入模板正文。
- AI 读取 README 后应按其中步骤接入；若缺少参数或令牌，先向开发者询问，不自行猜测。

### 2.8 `rule.md`
```md
# AI 写码人格与规则（每次写代码必须遵守）

## 使用方式
任何 AI 在为本项目编写或修改代码前，必须先阅读本文件，并在整个会话中遵守以下规则。

---

## 0. 开发前必读：需求文档优先（必须执行）

- **开发任何功能前，必须首先阅读 `PRODUCT_REQUIREMENTS.md`**，确认功能范围、技术选型、接口约定与验收标准。
- 若 `PRODUCT_REQUIREMENTS.md` 不存在或内容为空，必须先向用户索取或确认需求，不得在需求不明的情况下开始编码。
- 需求文档是一切开发决策的基准：技术选型、目录结构、接口设计、UI 实现均以其为准；与进度文件（`progress/`）冲突时，以需求文档为准并记录差异。

---

## 0.1 开发前必读：技术选型确认（必须执行）

- **开发任何功能前，必须阅读 `technology/TECH_STACK.md`**，确认当前项目的技术架构与各端技术栈：
  - **全栈项目**：确认统一技术栈（语言、框架、数据库、部署方式等）。
  - **前后端分离**：分别确认前端技术栈（框架、构建工具、组件库、路由等）与后端技术栈（语言、框架、ORM、鉴权方式等）。

## 1. 前后端联动与影响范围（必须执行）

### 1.1 修改前端代码时
- **必须**思考并主动检查：
  - 对**其他页面/组件**的影响：是否共用组件、共享状态、路由或布局；是否有引用处需要同步修改。
  - 对**后端**的影响：请求的 URL、方法、请求体/查询参数、响应结构是否与现有 API 一致；若前端期望变更接口契约，是否已与后端实现或 API 文档对齐，并同步修改后端或接口约定。
- 修改后若涉及接口形状变更，须在 API 层/后端做对应更新或记录在 `issue/`，避免前后端不一致。

### 1.2 修改后端代码时
- **必须**思考并主动检查：
  - 对**前端**的影响：接口路径、方法、请求/响应字段是否被前端使用；若有破坏性变更，须同步修改前端调用处及类型定义，或注明兼容方案与迁移步骤。
  - 对**其他接口或服务**的影响：是否被其他 API、定时任务或内部调用依赖；数据模型或权限逻辑变更是否影响现有功能。
- 修改后若涉及接口或数据契约变更，须同步更新前端调用与类型，或在 `progress/` / `issue/` 中记录待办。

### 1.3 建议动作
- 修改前可快速搜索/检索：相关 API 路径、组件名、类型定义、共享常量，再动手改。
- 一次改动若横跨前后端，尽量在同一轮对话中完成前后端一致修改，或明确列出未完成项并写入 `progress/CURRENT_PROGRESS.md` 的「存在问题」或 `issue/`。

## 2. 遵守进度：仅开发当前阶段及以下（必须执行）

- 开发时**必须**以 `progress/PHASES.md` 与 `progress/CURRENT_PROGRESS.md` 为准，**只开发或完善「当前进度」所在阶段及其之前阶段**的代码。
- 若当前阶段已完成任务拆解（即存在 `progress/阶段N/OVERVIEW.md`），**必须阅读该文件**，以其中列出的任务清单为本期开发的直接依据。
- 若 `progress/` 下**尚未存在当前阶段的子目录**（即尚未拆解任务），**必须先询问用户是否现在拆分任务，再开始开发**；不得在无任务拆解的情况下直接动手编码。
- 例如：若当前进度为**阶段二**，则只允许根据 `PHASES.md` 开发或完善**阶段二**和**阶段一**相关功能与代码；**不得**主动开发阶段三及以后在 `PHASES.md` 中定义的内容。
- 若用户明确要求提前做后续阶段，再按用户指示执行；否则一律以进度文件中的当前阶段为上限。

## 5. 开发完毕检查 issue 目录（必须执行）
```
