# 需求变更标准流程

> 适用场景：当前项目中，已经开发完成、正在联调、或已进入测试阶段的需求发生变更时，统一按本流程处理。

## 1. 目标

需求变更不能直接从技术修改开始。标准顺序应当是：

1. 先有产品变更说明
2. 再有标准差异文档
3. 再做影响分析
4. 最后进入开发、测试和文档回写

这样可以避免：
- 代码先改了，产品文档没更新
- 前后端和测试拿到的口径不一致
- 后续再看历史时无法判断变更来源

## 2. 适用范围

以下情况都按本流程执行：

- 已完成需求的小范围调整
- 联调阶段发现需要补字段、补规则、补校验
- 已有页面或接口的返回结构变更
- 权限、流程、状态规则发生变化
- 多个需求变更需要统一整理

## 3. 先区分变更类型

### 3.1 小型变更

适用：
- 新增少量字段
- 调整文案或校验
- 增加统计字段
- 不修改数据库结构
- 不改变主流程

建议产物：
- 一份产品变更说明
- 一份标准 diff 文档
- 更新 `requirements/CURRENT.md`

### 3.2 中大型变更

适用：
- 改接口契约
- 改流程
- 改权限模型
- 改数据库结构
- 同时影响多个模块

建议产物：
- 保留原始需求文档
- 新增新版产品文档或正式变更说明
- 一份标准 diff 文档
- 更新 `requirements/CURRENT.md`
- 再进入影响分析与拆批开发

### 3.3 多项变更同时发生

处理原则：
- 先汇总讨论
- 再拆批执行

不要：
- 一条一条零散直接开发
- 多个高风险变更混成一个包一起改

推荐拆批方式：
- 同一模块、同一接口、同一页面的改动放一起
- 高风险改动单独成批
- 字段追加类低风险变更可合并

## 4. 标准流程

### 步骤 1：产品先提交变更说明

产品变更应先落文档，不直接口头驱动技术改动。

建议至少包含：
- 变更标题
- 变更模块
- 当前行为
- 目标行为
- 变更原因
- 来源依据
- 验收口径

原始产品资料统一归档到 `docs/`。

### 步骤 2：保留原文，不覆盖旧版本

要求：
- 原始 PRD、接口文档、技术文档继续保留
- 新版文档或变更说明进入 `docs/`
- 不直接覆盖旧文档导致无法回溯

### 步骤 3：生成标准差异文档

差异文档放在：
- `requirements/diffs/DIFF_*.md`

结构必须符合 `AI_PRD_GUIDE.md §0.1`：

1. 一、增量
2. 二、修改
3. 三、删除

说明：
- 增量：原来没有，现在新增
- 修改：原有内容发生变化
- 删除：原有内容不再保留；如果没有删除项，必须明确写“无实质删除”

### 步骤 4：更新当前需求入口

更新：
- `requirements/CURRENT.md`

作用：
- 告诉前端、后端、测试当前最新变更是什么
- 让执行入口统一，不靠口头同步

### 步骤 5：做影响分析

影响分析不能只看代码，要同时看：
- 前端
- 后端
- 测试
- 接口文档
- 进度文档
- 权限与审计
- 历史数据与兼容性

建议至少明确：
- 改哪些接口
- 改哪些字段
- 是否影响旧前端
- 是否影响旧测试用例
- 是否需要补文档

### 步骤 6：确认后再开发

开发顺序建议：
1. 后端 / 前端按影响范围实施
2. 补自动化测试或测试用例
3. 回写接口文档
4. 更新进度文档

### 步骤 7：测试与回归

测试更新时，必须基于：
- `requirements/CURRENT.md`
- 最新 `requirements/diffs/DIFF_*.md`

测试项按三类拆分：
- 新增测试点
- 修改旧用例
- 删除或废弃旧用例

同时补回归验证：
- 旧逻辑是否仍正常
- 兼容要求是否满足

## 5. 本项目推荐落地产物

一次规范的需求变更，至少应包含：

1. `docs/` 下的原始变更依据
2. `requirements/diffs/DIFF_*.md`
3. `requirements/CURRENT.md` 的变更入口更新
4. 对应代码修改
5. 测试更新
6. 接口文档或执行基线文档更新

## 6. 角色分工建议

### 产品

- 先提交变更说明
- 明确“当前行为”和“目标行为”
- 明确验收口径

### 后端

- 根据 diff 做接口和逻辑影响分析
- 补测试
- 回写接口文档

### 前端

- 不从头开发
- 只做本次变更直接影响的增量适配

### 测试

- 不从头重写测试
- 按 diff 的“增量 / 修改 / 删除”更新用例

## 7. 最容易出问题的错误做法

以下做法不规范：

- 先改代码，后补产品文档
- 只口头说变更，不留文档
- 接口已经改了，但 `requirements/CURRENT.md` 没更新
- 前端只看代码，不看 diff
- 测试不知道哪些是新增，哪些是修改，哪些应废弃

## 8. 执行原则

一句话总结：

需求变更要先从产品文档出发，再形成标准 diff，再进入技术实现。

技术修改只是流程的后半段，不是起点。
