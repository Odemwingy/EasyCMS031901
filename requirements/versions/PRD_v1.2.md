## 2. 产品背景与阶段目标

### 2.1 整体产品背景

EasyCMS 是一套面向国产 IFE 场景的地面内容管理平台，统一承载内容主数据管理、分类编排、审核签核、周期管理、配置治理、导出发布和质量追踪，服务内部员工与企业客户两类用户。

### 2.2 当前开发目标

当前仅进入**阶段一：后台管理一期**开发，目标是先交付平台治理底座，支撑后续内容中心、周期管理、配置中心、导出发布等业务模块接入。

阶段一要解决的问题：
- 统一用户账号生命周期管理
- 建立角色与权限治理能力
- 通过菜单树支撑页面级和操作级权限
- 对关键后台操作形成可查询的审计日志
- 提供登录认证与基础后台框架

## 3. 当前开发范围

### 3.1 阶段一 In Scope

本阶段正式开发范围仅包括以下模块：
- 登录 / 登出 / 强制改密
- 用户管理
- 角色管理
- 菜单管理
- 权限配置
- 审计日志

### 3.2 阶段一 Out of Scope

以下能力当前不进入正式开发：
- 忘记密码 / 邮件激活

- 数据范围权限过滤逻辑
- 互斥角色约束
- 批量导入用户
- 消息中心
- 通知配置
- 系统参数页

说明：
- `app/frontend/` 中已存在的通知配置页、系统参数页属于 UI/原型产物，不视为阶段一后续联调交付范围。
- 数据范围字段允许预留，但本阶段不落地组织级/项目级过滤逻辑。

## 4. 用户角色与使用对象

当前阶段重点涉及以下角色：
- 系统管理员：用户、角色、权限、菜单、审计的最高管理者
- 项目管理员：查看项目范围内用户和日志信息
- 内容运营：登录后看到被授权菜单，不得越权
- 内容服务商：企业客户侧用户，后续会使用组织与项目范围字段
- 审核人员：后续业务模块中的审批与签核角色
- 只读用户：只查看，不允许任何写操作

## 5. 阶段一功能需求

### 5.1 登录与账号安全

功能要求：
- 采用平台自建账号密码体系，不接 SSO / LDAP
- 用户通过用户名和密码登录
- 登录成功后进入后台首页
- 首次登录或管理员重置密码后，必须强制修改密码
- 停用或锁定账号不可登录

密码规则：
- 最小长度 8 位
- 必须包含大写字母、小写字母、数字
- 连续失败 5 次后自动锁定
- 支持管理员手动解锁，或 30 分钟后自动解锁
- 会话超时 8 小时

异常提示：
- 用户名或密码错误
- 账号已停用
- 账号已锁定
- 登录已过期

验收要点：
- 正确账号可登录
- 连续失败 5 次后账号锁定
- 首次登录必须改密
- 会话超时后返回登录页

### 5.2 用户管理

用户管理需要支持：
- 用户列表分页查询
- 关键字搜索（用户名 / 姓名）
- 组织、角色、状态、用户类型筛选
- 新建用户
- 编辑用户
- 启用 / 停用用户
- 解锁用户
- 重置密码
- 查看用户详情

用户核心字段：
- `id`
- `username`
- `name`
- `user_type`
- `email`
- `mobile`
- `org_id`
- `project_ids`
- `role_ids`
- `status`
- `must_change_password`
- `last_login_at`
- `created_at`
- `updated_at`
- `remark`

业务规则：
- 用户名全局唯一
- 用户创建后用户名不可修改
- 至少保留一个系统管理员账号处于启用状态
- 用户可以绑定多个角色，最终权限按并集计算，冲突时以禁止优先
- 停用后立即禁止登录，但保留历史审计记录

### 5.3 角色管理

角色管理需要支持：
- 角色列表分页查询
- 新建角色
- 编辑角色
- 复制角色
- 启用 / 停用角色
- 删除角色
- 查看角色详情
- 查看绑定用户

建议预置角色：
- 系统管理员
- 项目管理员
- 内容运营
- 内容服务商
- 审核人员
- 发布人员
- 配置管理员
- 只读用户
- 支持排障人员

角色字段：
- `id`
- `name`
- `code`
- `description`
- `data_scope`
- `status`
- `is_system_preset`
- `created_at`
- `updated_at`

业务规则：
- 系统预置角色不可删除，但允许复制
- 已绑定用户的角色不可直接删除
- 停用角色时若存在活跃用户，必须阻止操作
- 角色编码创建后不可修改
- 数据范围字段本阶段保留，默认固定为 `all`

### 5.4 菜单管理

菜单管理是权限控制的基础数据源，阶段一必须支持：
- 获取完整菜单树
- 新建菜单节点
- 编辑菜单节点
- 启用 / 停用菜单节点
- 删除菜单节点

节点类型：
- 目录
- 菜单项
- 按钮

核心字段：
- `id`
- `parent_id`
- `type`
- `name`
- `permission`
- `route_path`
- `component`
- `icon`
- `sort`
- `status`
- `remark`

业务规则：
- 权限标识全局唯一
- 权限标识格式统一为 `模块:资源:动作`
- 按钮节点只能挂在菜单项下
- 菜单树最多支持 3 级
- 停用父节点时子节点同步停用
- 已被角色引用的菜单节点不允许删除
- 菜单节点创建后 `permission` 不可修改

### 5.5 权限配置

权限配置以菜单树为数据源，支持：
- 页面级权限
- 操作级权限
- 只读策略

权限模型：
- 模块权限：一级目录是否可见
- 页面权限：菜单项是否可访问
- 操作权限：按钮动作是否可执行
- 数据范围：字段保留，不落地过滤逻辑

权限规则：
- 用户最终权限 = 所有角色权限的并集
- 冲突时禁止优先
- 未授权菜单在导航中隐藏
- 直接访问无权限页面时返回无权限页
- 无操作权限按钮不展示
- 后端必须在每次请求时校验权限
- 权限变更在下次请求时生效，不强制踢下线
- 菜单节点停用后，对应角色权限自动失效

### 5.6 审计日志

阶段一需要落地以下审计类型：
- 用户权限日志
- 角色权限日志
- 菜单管理日志
- 登录日志

支持能力：
- 日志列表查询
- 多条件筛选
- 日志详情查看

日志字段：
- `id`
- `log_type`
- `operator_id`
- `operator_name`
- `action`
- `object_type`
- `object_id`
- `object_name`
- `before_value`
- `after_value`
- `result`
- `fail_reason`
- `source_page`
- `request_id`
- `project_id`
- `created_at`

记录规则：
- 所有写操作必须记录日志
- 失败操作也必须记录
- 日志不可编辑、不可删除
- 登录成功、登录失败、账号锁定都要记录
- 默认保留 90 天

## 6. 阶段一接口执行基线

统一约定：
- 接口前缀：`/api/v1`
- 鉴权方式：Bearer Token
- 响应结构：
  - `code`
  - `message`
  - `data`
  - `timestamp`

公共错误码：
- `0` 成功
- `1001` 参数校验失败
- `1002` 未登录或 token 失效
- `1003` 无权限
- `1004` 资源不存在
- `1005` 业务规则拦截
- `5000` 服务异常

阶段一主要接口范围：
- 认证
  - `POST /api/v1/auth/login`
  - `POST /api/v1/auth/logout`
  - `POST /api/v1/auth/change-password`
  - `GET /api/v1/auth/me`
  - `GET /api/v1/auth/menus`
- 用户管理
  - `GET /api/v1/admin/users`
  - `GET /api/v1/admin/users/{id}`
  - `POST /api/v1/admin/users`
  - `PUT /api/v1/admin/users/{id}`
  - `PATCH /api/v1/admin/users/{id}/status`
  - `PATCH /api/v1/admin/users/{id}/unlock`
  - `POST /api/v1/admin/users/{id}/reset-password`
- 角色管理
  - `GET /api/v1/admin/roles`
  - `GET /api/v1/admin/roles/{id}`
  - `POST /api/v1/admin/roles`
  - `PUT /api/v1/admin/roles/{id}`
  - `POST /api/v1/admin/roles/{id}/copy`
  - `PATCH /api/v1/admin/roles/{id}/status`
  - `DELETE /api/v1/admin/roles/{id}`
  - `GET /api/v1/admin/roles/{id}/permissions`
  - `PUT /api/v1/admin/roles/{id}/permissions`
- 菜单管理
  - `GET /api/v1/admin/menus/tree`
  - `POST /api/v1/admin/menus`
  - `PUT /api/v1/admin/menus/{id}`
  - `PATCH /api/v1/admin/menus/{id}/status`
  - `DELETE /api/v1/admin/menus/{id}`
- 审计日志
  - `GET /api/v1/admin/audit-logs`
  - `GET /api/v1/admin/audit-logs/{id}`

枚举基线：
- 用户状态：`1 启用`、`2 停用`、`3 锁定`、`4 未激活`
- 用户类型：`1 内部员工`、`2 企业客户`
- 角色状态：`1 启用`、`2 停用`
- 菜单类型：`1 目录`、`2 菜单项`、`3 按钮`

## 8. 当前阶段验收标准

前端：
- 登录页、登出逻辑、路由守卫可用
- 后台管理页面完成真实接口对接
- 基于用户菜单实现导航与按钮权限控制

后端：
- 完成认证、用户、角色、菜单、审计日志的基础可用接口
- Bearer Token 鉴权可用
- 接口级权限拦截可用
- 审计日志覆盖关键后台写操作

整体：
- 阶段一前后端契约对齐
- 后台管理一期可完成登录、用户/角色/菜单/权限治理与审计查询闭环
