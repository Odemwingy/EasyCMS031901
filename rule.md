# AI 写码人格与规则（每次写代码必须遵守）

## 使用方式
任何 AI 在为本项目编写或修改代码前，必须先阅读本文件，并在整个会话中遵守以下规则。

---

## 0. 开发前必读：需求文档优先（必须执行）

- **开发任何功能前，必须首先阅读 `requirements/versions/PRD_v1.2.md`**，确认功能范围、技术选型、接口约定与验收标准。
- 若 `requirements/versions/PRD_v1.2.md` 不存在或内容为空，必须先向用户索取或确认需求，不得在需求不明的情况下开始编码。
- 需求文档是一切开发决策的基准：技术选型、目录结构、接口设计、UI 实现均以其为准；与进度文件（`progress/`）冲突时，以需求文档为准并记录差异。

---

## 0.1 开发前必读：技术选型确认（必须执行）

- **开发任何功能前，必须阅读 `technology/TECH_STACK.md`**，确认当前项目的技术架构与各端技术栈：
  - **全栈项目**：确认统一技术栈（语言、框架、数据库、部署方式等）。
  - **前后端分离**：分别确认前端技术栈（框架、构建工具、组件库、路由等）与后端技术栈（语言、框架、ORM、鉴权方式等）。
- 所有代码**必须严格遵循** `technology/TECH_STACK.md` 中记录的技术决策，**不得擅自引入未记录的框架、库或运行时**。
- 若需要新增或变更技术选型，必须先更新 `technology/TECH_STACK.md`，在 `memory/` 写入变更记录，再开始编码。
- 若 `technology/TECH_STACK.md` 不存在或内容为空，必须先向用户确认技术选型，不得自行假设。

## 1. 前后端联动与影响范围（必须执行）

### 1.1 修改前端代码时
- **必须**思考并主动检查：
  - 对**其他页面/组件**的影响：是否共用组件、共享状态、路由或布局；是否有引用处需要同步修改。
  - 对**后端**的影响：请求的 URL、方法、请求体/查询参数、响应结构是否与现有 API 一致；若前端期望变更接口契约，是否已与后端实现或 API 文档对齐，并同步修改后端或接口约定。
- 修改后若涉及接口形状变更，须在 API 层/后端做对应更新或记录在 `issue/`，避免前后端不一致。

### 1.2 修改后端代码时
- **必须**思考并主动检查：
  - 对**前端**的影响：接口路径、方法、请求/响应字段是否被前端使用；若有破坏性变更，须同步修改前端调用处及类型定义，或注明兼容方案与迁移步骤。
  - 对**其他接口或服务**的影响：是否被其他 API、定时任务或内部调用依赖；数据模型或权限逻辑变更是否影响现有功能。
- 修改后若涉及接口或数据契约变更，须同步更新前端调用与类型，或在 `progress/` / `issue/` 中记录待办。

### 1.3 建议动作
- 修改前可快速搜索/检索：相关 API 路径、组件名、类型定义、共享常量，再动手改。
- 一次改动若横跨前后端，尽量在同一轮对话中完成前后端一致修改，或明确列出未完成项并分别写入 `progress/frontend.md` / `progress/backend.md` 的「存在问题」列，并同步更新 `progress/CURRENT_PROGRESS.md` 阻塞项或记录在 `issue/`。

## 2. 遵守进度：仅开发当前阶段及以下（必须执行）

### 2.1 当前阶段限制
- 开发时**必须**以 `progress/PHASES.md` 与 `progress/CURRENT_PROGRESS.md` 为准，**只开发或完善「当前进度」所在阶段及其之前阶段**的代码。
- 若当前阶段已完成任务拆解（即存在 `progress/阶段N/OVERVIEW.md`），**必须阅读该文件**，以其中列出的任务清单为本期开发的直接依据。
- 若 `progress/` 下**尚未存在当前阶段的子目录**（即尚未拆解任务），**必须先询问用户是否现在拆分任务，再开始开发**；不得在无任务拆解的情况下直接动手编码。
- 例如：若当前进度为**阶段二**，则只允许根据 `PHASES.md` 开发或完善**阶段二**和**阶段一**相关功能与代码；**不得**主动开发阶段三及以后在 `PHASES.md` 中定义的内容。
- 若用户明确要求提前做后续阶段，再按用户指示执行；否则一律以进度文件中的当前阶段为上限。

### 2.2 并行开发时的进度文件规范（必须执行）
- `progress/` 目录下存在**主控总览**和**各端子进度文件**，**各端只写自己的子文件**，严禁多端同时修改同一文件导致冲突：
  - **开发前端**时，读取并更新 `progress/frontend.md`；
  - **开发后端**时，读取并更新 `progress/backend.md`；
  - **产出 UI 设计稿**时，读取并更新 `progress/ui.md`；
  - **编写/执行测试**时，读取并更新 `progress/tests.md`。
- `progress/CURRENT_PROGRESS.md`（主控总览）**只在以下时机更新**：阶段切换、重大里程碑完成，或各端均完成本阶段时；更新时只修改「各端状态总览」表和「阶段切换记录」，不写各端详细任务列表。
- `progress/PHASES.md` 只在阶段定义或需求发生变更时更新。
- 开发前**必须**检查进度；若 `progress/` 下缺少任务拆解（子进度文件为空），必须询问用户是否需要先拆分任务，再开始编码。

## 3. UI 还原：以 ui 目录与设计数据为准（必须执行）

- 开发界面代码时**必须**查看 `ui/` 目录，以**高还原度**为准实现。
- 若 `ui/` 下提供了 **MCP** 或 **Figma** 的链接/接入方式，**优先以这些数据为准**（尺寸、间距、颜色、组件结构等），再落成页面与组件代码。
- 未提供 MCP 或 Figma 时，以 `ui/` 内的设计稿、说明或 token 导出为准，保证实现与设计一致。

---

## 4. 产出位置：代码与产品产物均在 app 目录下

- 本项目的**代码**与**产品产物**均生成在 **`app/`** 目录下。
- 前端页面、后端 API、组件、样式、配置等应用相关产出一律放在 `app/` 或其子目录中，不散落在项目根目录或其他约定外的位置。

---

## 6. 其他通用规则（可随项目补充）

- 遵守 `requirements/versions/PRD_v1.2.md` 与 `AI_PRD_GUIDE.md` 中的技术选型、目录约定与产出要求。
- 关键逻辑（鉴权、权限、审计日志、支付等）须有中文注释说明意图与风险点。
- 不提交敏感信息（密码、密钥、Token）进仓库；配置通过环境变量或 `.env`（且 `.env` 不提交）。
- 若发现需求歧义或实现冲突，先与用户确认，不擅自二选一。

---

## 7. memory 记录规则（必须执行）

发生以下任一情况后，**AI 必须主动**在 `memory/` 目录下新增或追加当日记忆文件（`YYYY-MM-DD.md`）：

1. **修改了基础配置文件**：对 `AI_PRD_GUIDE.md` 或 `rule.md` 做了实质性内容修改（新增/删除/调整规则、模板或约定）。
2. **修改了需求文档**：`requirements/versions/PRD_v1.2.md` 发生功能范围、验收标准、接口约定或技术选型的实质性变更。
3. **变更了技术架构**：`technology/TECH_STACK.md` 发生变更（技术栈选型、目录结构约定、数据库 Schema、API 规范等重大调整）。
4. **完成重要里程碑**：一个完整阶段交付、重大功能上线或部署。
5. **重大 Bug 修复或重构**：影响多模块或多页面的修复、方案重写或架构重构。

记忆文件格式参照 `memory/README.md`，内容需包含：变更类型、变更摘要、变更详情、影响范围、当前项目状态。
