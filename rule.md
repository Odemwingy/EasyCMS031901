# AI 写码人格与规则（每次写代码必须遵守）

## 使用方式
任何 AI 在为本项目编写或修改代码前后，**必须**阅读本文件，并在整个会话中遵守以下规则。

---

## 1. 前后端联动与影响范围（必须执行）

### 1.1 修改前端代码时
- **必须**思考并主动检查：
 - 对**其他页面/组件**的影响：是否共用组件、共享状态、路由或布局；是否有引用处需要同步修改。
 - 对**后端**的影响：请求的 URL、方法、请求体/查询参数、响应结构是否与现有 API 一致；若前端期望变更接口契约，是否已与后端实现或 API 文档对齐，并同步修改后端或接口约定。
- 修改后若涉及接口形状变更，须在 API 层/后端做对应更新或记录在 `issue/`，避免前后端不一致。

### 1.2 修改后端代码时
- **必须**思考并主动检查：
 - 对**前端**的影响：接口路径、方法、请求/响应字段是否被前端使用；若有破坏性变更，须同步修改前端调用处及类型定义，或注明兼容方案与迁移步骤。
 - 对**其他接口或服务**的影响：是否被其他 API、定时任务或内部调用依赖；数据模型或权限逻辑变更是否影响现有功能。
- 修改后若涉及接口或数据契约变更，须同步更新前端调用与类型，或在 `progress/` / `issue/` 中记录待办。

### 1.3 建议动作
- 修改前可快速搜索/检索：相关 API 路径、组件名、类型定义、共享常量，再动手改。
- 一次改动若横跨前后端，尽量在同一轮对话中完成前后端一致修改，或明确列出未完成项并写入 `progress/CURRENT_PROGRESS.md` 的「存在问题」或 `issue/`。

---

## 2. 遵守进度：仅开发当前阶段及以下（必须执行）
- 开发时**必须**以 `progress/PHASES.md` 与 `progress/CURRENT_PROGRESS.md` 为准，**只开发或完善「当前进度」所在阶段及其之前阶段**的代码。
- 例如：若当前进度为**阶段二**，则只允许根据 `PHASES.md` 开发或完善**阶段二**和**阶段一**相关功能与代码；**不得**主动开发阶段三及以后在 `PHASES.md` 中定义的内容。
- 若用户明确要求提前做后续阶段，再按用户指示执行；否则一律以进度文件中的当前阶段为上限。

---

## 3. UI 还原：以 ui 目录与设计数据为准（必须执行）
- 开发界面代码时**必须**查看 `ui/` 目录，以**高还原度**为准实现。
- 若 `ui/` 下提供了 **MCP** 或 **Figma** 的链接/接入方式，**优先以这些数据为准**（尺寸、间距、颜色、组件结构等），再落成页面与组件代码。
- 未提供 MCP 或 Figma 时，以 `ui/` 内的设计稿、说明或 token 导出为准，保证实现与设计一致。

---

## 4. 产出位置：代码与产品产物均在 app 目录下
- 本项目的**代码**与**产品产物**均生成在 **`app/`** 目录下。
- 前端页面、后端 API、组件、样式、配置等应用相关产出一律放在 `app/` 或其子目录中，不散落在项目根目录或其他约定外的位置。

---

## 5. 开发完毕检查 issue 目录（必须执行）
- **每次开发完毕后**，检查 `issue/` 目录。
- `issue/` 下的文件使用 **Word 文档**（`.docx`）；每个 issue 的状态通过**文件名或文档内标注**体现，取值为：**已解决**、**未解决**、**待定**（例如文件名：`已解决-xxx.docx` / `未解决-xxx.docx` / `待定-xxx.docx`）。
- 检查时**忽略**状态为「已解决」的 issue，**仅针对「未解决」**的 issue 逐一向开发者询问：该 issue 是否已经解决？
- 若开发者回复表示已解决，则**将状态更新为「已解决」**（如通过重命名文件为 `已解决-原标题.docx` 或在文档内更新状态，以项目约定为准）。

---

## 6. 其他通用规则（可随项目补充）
- 遵守 `PRODUCT_REQUIREMENTS.md` 与 `AI_PRD_GUIDE.md` 中的技术选型、目录约定与产出要求。
- 关键逻辑（鉴权、权限、审计日志、支付等）须有中文注释说明意图与风险点。
- 不提交敏感信息（密码、密钥、Token）进仓库；配置通过环境变量或 `.env`（且 `.env` 不提交）。
- 若发现需求歧义或实现冲突，先与用户确认，不擅自二选一。

---

*本项目可根据需要在本文件中继续追加规则，AI 以最新内容为准。*
