# 环境与数据保护规则

> 适用场景：当前项目中，AI、前端、后端、测试在联调、修复 Bug、执行回归、操作 GitHub Issue / PR 时，统一遵守本规则。

## 1. 目标

本规则用于降低以下风险：

- 误清空真实开发库
- 用测试数据污染共享环境
- 将 Token、密钥、凭证误写入仓库或评论
- 因编码问题把评论、Issue、测试数据写成乱码
- 在未隔离的环境中执行高风险命令

一句话原则：

共享环境优先保护，测试默认隔离，机密不入仓库，中文文本走 UTF-8 文件链路。

## 2. 数据库与环境保护

### 2.1 测试默认使用隔离环境

- 自动化测试默认使用隔离数据库
- 优先使用内存库、测试专用库或临时数据库
- 不允许把真实开发 MySQL 作为测试默认目标

### 2.2 禁止重置真实开发库

以下命令不得对真实开发库执行：

- `migrate:fresh`
- `db:wipe`
- 任何清空业务表的 SQL
- 任何覆盖式重建初始化数据的高风险操作

如确需执行高风险数据库操作，必须先确认：

- 当前目标库不是共享开发库
- 当前操作不会影响测试、联调或历史数据

### 2.3 共享环境只做最小必要验证

如果必须在真实开发环境验证：

- 只创建最小量临时数据
- 验证完成后立即清理
- 不影响现有账号、角色、菜单、审计日志的真实业务数据

## 3. 测试数据规则

### 3.1 不直接用命令行内联中文造测试数据

禁止使用以下方式直接写中文测试数据：

- PowerShell 内联中文参数
- 容器命令行里直接拼中文文本
- 临时 SQL / tinker 命令中直接写中文字符串

原因：

- Windows / PowerShell / 容器链路下容易编码损坏
- 中文可能在入库前就变成 `????`

### 3.2 中文测试数据的推荐方式

如果确实需要中文文本：

- 优先通过真实 HTTP 接口创建
- 或使用 UTF-8 编码的 Seeder / 脚本文件
- 若只是验证逻辑，不依赖中文时优先用英文占位值

## 4. 凭证与机密保护

### 4.1 Token 只放本地 `config/`

允许：

- `config/token.txt`
- `config/github.local.env`
- 其他本地未纳入 Git 的配置文件

禁止：

- 提交到仓库
- 写到文档正文
- 写到 PR 描述、Issue 评论
- 在聊天中长期明文保留

### 4.2 使用后优先撤销暴露凭证

如 Token 曾在聊天、日志、截图中明文出现，应视为泄露：

- 尽快在 GitHub 撤销
- 重新生成最小权限新 Token

## 5. GitHub 评论与 Issue 更新的编码规则

### 5.1 中文评论不要直接走命令行内联正文

如果要在 GitHub Issue / PR 评论中写中文：

- 不要把中文正文直接拼在 PowerShell 或 curl 命令里

### 5.2 推荐使用 UTF-8 文件中转

推荐流程：

1. 先把评论正文写入 UTF-8 文件
2. 从文件中读取文本
3. 再通过 GitHub API 提交或更新评论

这样可以避免：

- 评论变成 `????`
- Markdown 内容被命令行转义破坏

## 6. 分支与修复安全

- 不直接在 `main` 上修复问题
- Bug 修复走独立 `codex/*` 分支
- 需求变更先走 `requirements/CHANGE_PROCESS.md`
- 修复完成先发 PR，再转测试回归

## 7. 风险动作前的检查项

执行以下动作前，必须再次确认：

- 数据迁移
- 数据清理
- 真实环境造数
- Token 写评论
- 批量更新角色 / 权限 / 菜单

至少确认：

1. 当前环境是否共享
2. 当前数据是否可恢复
3. 当前操作是否会影响测试和联调
4. 当前文本是否有编码风险

## 8. 执行原则

一句话总结：

任何会影响共享环境、真实数据、机密凭证、中文文本编码的操作，都优先选择更安全、更可回退的方式。
