# 需求变更模板

用于已经开发完成或正在联调的需求发生变更时，统一提交变更说明，减少范围漂移和返工。

## 使用规则

1. 已上线、已开发完成、已联调通过的功能发生变更时，优先使用本模板。
2. 先提交变更说明，再做影响分析，再进入开发，不直接口头改需求。
3. 如果变更会影响接口、数据库结构、权限模型或历史数据，必须明确写出兼容要求。
4. 如果暂时没有完整信息，至少先写清楚“当前行为”“目标行为”“验收口径”。
5. 变更分析时，默认必须检查接口、数据、权限、流程、审计、兼容、测试、文档 8 类约束。

## 推荐流程

1. 提交一条变更单
2. 评估影响范围
3. 确认实施方案
4. 开发与回归测试
5. 更新文档与进度
6. 决定是否提交 / 推送

## 变更分析默认检查项

### 1. 接口契约约束

- 请求字段是否变化
- 必填 / 选填是否变化
- 响应字段、错误码、提示文案是否变化
- 现有前端请求是否还能兼容

### 2. 数据约束

- 是否影响表结构、字段类型、默认值、枚举值
- 是否需要迁移历史数据
- 旧数据在新规则下是否仍合法
- 空值、缺省值、非法值如何处理

### 3. 权限约束

- 是否影响菜单权限或按钮权限
- 是否影响角色能力边界
- 是否影响 `system_admin`、预置角色、只读角色
- 是否会导致关键页面无法访问

### 4. 流程约束

- 是否影响上下游流程
- 是否影响创建、编辑、删除、状态切换、登录、权限配置等链路
- 是否存在多步骤流程中的状态不一致问题

### 5. 审计约束

- 是否需要新增或调整审计日志写入
- 成功分支和失败分支是否都要记录
- 是否涉及安全、权限、账号、敏感数据变更

### 6. 兼容性约束

- 是否需要兼容旧前端、旧请求、旧数据
- 是否允许 breaking change
- 是否需要双逻辑过渡期

### 7. 测试约束

- 需要补哪些回归测试
- 需要覆盖哪些边界场景
- 是否涉及历史数据、权限不足、并发重复提交等场景

### 8. 文档与进度约束

- 是否需要更新 PRD / 接口文档 / 技术文档
- 是否需要更新 `progress`
- 是否需要补 `changelog` 或 `memory`

## 最小填写示例

```md
变更标题：编辑用户时状态改为非必填

变更模块：用户管理 - 编辑用户

当前行为：编辑用户接口必须传 status，否则返回 “The status field is required.”

目标行为：编辑用户接口不传 status 时保留原值，不报错。

变更原因：前端现有编辑请求未传 status，且接口文档未要求该字段为必填。

来源依据：
- PRD：阶段一后台管理 PRD
- 接口文档：编辑用户接口
- 技术文档：无额外限制
- 会议结论 / Issue：联调问题修复

影响范围：
- 后端：UpdateUserRequest、UserController
- 前端：无需修改
- 数据库：无
- 权限：无
- 测试：补充编辑用户回归测试

兼容要求：兼容现有前端请求，不影响已传 status 的请求。

风险点：需要确认未传 status 时不会把状态写空。

约束条件：
- 接口约束：编辑用户接口响应结构不变
- 数据约束：不修改表结构，不修改历史数据
- 权限约束：不影响原有编辑用户权限
- 流程约束：不影响状态切换接口
- 审计约束：编辑用户日志逻辑保持不变
- 兼容约束：兼容旧前端不传 status 的请求

边界场景：
- 空值 / 缺省值：status 缺省时保留原值
- 非法值：传非法 status 仍返回校验错误
- 历史数据：已有用户状态不发生变更
- 权限不足：无编辑权限仍返回无权限
- 并发 / 重复提交：保持现有行为
- 极端路径：编辑其他字段成功时不应误改状态

不在本次范围：
- 不调整用户状态切换接口
- 不调整前端请求结构

验收口径：
1. 前端不传 status 可以编辑成功
2. 原用户状态保持不变
3. 已传 status 的编辑请求行为不变
```

## 我收到变更单后的默认动作

1. 先做影响分析，不直接改代码
2. 标出会影响的接口、表结构、权限、测试
3. 补齐约束条件和边界场景
4. 你确认后再实施
5. 实施完成后补测试、补文档、补进度

## 备注

- 如果你明确说“只改后端”或“前端不能动”，我会按这个约束执行。
- 如果变更与现有 PRD / 接口文档冲突，我会先指出冲突点，再给建议方案。
- 对已完成需求的变更，默认不直接修改数据库结构，除非变更单里明确要求。
