# Issue 处理标准流程

> 适用场景：当前项目中，测试、产品、前端、后端围绕 GitHub Issue 进行缺陷处理、回归流转和需求分流时，统一按本流程执行。

## 1. 目标

本流程用于统一以下行为：

1. 读取并处理分配到个人名下的 GitHub Issue
2. 区分“真实缺陷”和“需求变更”
3. 统一前端、后端、测试之间的流转方式
4. 避免直接在 `main` 上修复并造成流程失控

一句话原则：

Issue 先分类，再修复；修复走分支和 PR；修完先转测试回归，不直接关闭。

## 2. 基本原则

- GitHub Issue 是唯一问题入口
- 所有处理动作以 Issue 内容为准，不以口头描述为准
- 修复代码不直接提交到 `main`
- 一条 Issue 对应一个独立 `codex/*` 分支
- 修复完成后先发 PR、回写 Issue、转给测试，不直接关闭
- 如果 Issue 本质是需求变更，不直接修代码，先走 `requirements/CHANGE_PROCESS.md`
- 涉及共享环境、真实数据、Token、中文评论时，同时遵守 `docs/ENGINEERING_GUARDRAILS.md`
- 不同角色接手任务前，建议先看 `docs/ROLE_ENTRYPOINTS.md`

## 3. Issue 分类

收到 Issue 后，先判断属于哪一类：

### 3.1 Bug

满足以下特征之一时，按 Bug 处理：

- 当前实现与 PRD / 接口文档 / 页面预期不一致
- 已有接口、页面、交互出现异常
- 自动化回归或手工测试出现失败

### 3.2 Requirement Change

满足以下特征之一时，按需求变更处理：

- 当前实现本身符合旧文档，但产品希望改成新的行为
- 接口字段、页面文案、流程规则、权限规则发生变化
- 测试反馈“要改成另一种表现”，但当前不是实现错误

Requirement Change 不直接修代码，统一转：

- `requirements/CHANGE_PROCESS.md`

### 3.3 Question / Environment

满足以下情况时，先不直接开发：

- 测试无法判断是否为缺陷
- 是环境、代理、账号、权限、数据脏污导致的问题
- 证据不足，无法直接复现

## 4. 标准处理流程

### 步骤 1：读取 Issue

至少确认以下内容：

- Issue 标题
- 描述
- 复现步骤
- 实际结果
- 期望结果
- 标签
- assignee
- 评论区补充信息

### 步骤 2：做分类判断

输出明确结论：

- 是 Bug
- 是需求变更
- 是环境 / 问题待澄清

### 步骤 3：如果是需求变更，停止直接开发

按下面顺序处理：

1. 产品补变更说明
2. 生成标准 diff
3. 更新 `requirements/CURRENT.md`
4. 再进入前端 / 后端 / 测试影响分析

参照：

- `requirements/CHANGE_PROCESS.md`

### 步骤 4：如果是 Bug，先复现和定位

至少确认：

- 是否能稳定复现
- 根因在前端还是后端
- 是否需要数据修复
- 是否涉及接口、权限、审计、兼容性

### 步骤 5：创建独立修复分支

规则：

- 从最新 `main` 切新分支
- 分支名使用 `codex/*`
- 一条 Issue 一条分支

推荐命名：

- `codex/fix-issue-16-role-permission-readback`
- `codex/fix-user-search-500`
- `codex/fix-login-redirect`

### 步骤 6：修复代码并补测试

要求：

- 只改本次 Issue 直接影响的代码
- 不顺手改无关模块
- 至少补一条能覆盖该问题的回归测试
- 如需更新接口文档 / 执行基线文档，应同步更新

### 步骤 7：提交、推分支、发 PR

标准顺序：

1. 提交到当前 `codex/*` 分支
2. 推送远端分支
3. 发起 PR 到 `main`

禁止：

- 直接 push 到 `main`

### 步骤 8：回写 Issue

修复完成后必须在 Issue 下留言，至少包含：

- 根因
- 修复内容
- 验证结果
- 提交号
- PR 链接

### 步骤 9：转给测试回归，不直接关闭

规则：

- 修复完成后，Issue 保持 `open`
- assignee 从开发转给测试
- 由测试基于修复分支或 PR 回归验证

测试回归通过后，再决定是否关闭。

## 5. 测试回归流程

测试接到转派后的处理方式：

1. 基于开发给出的分支 / PR 回归
2. 验证原问题是否修复
3. 验证是否引入回归
4. 在 Issue 下补回归结果

### 5.1 回归通过

建议动作：

- 评论说明“已验证通过”
- 根据团队习惯关闭 Issue 或转回开发收尾

### 5.2 回归失败

建议动作：

- 在原 Issue 继续补充失败现象
- 重新指派回对应开发
- 不新建重复 Issue，除非是另一个独立问题

## 6. 前端 / 后端 / 测试角色分工

### 前端

- 只处理前端侧根因
- 不擅自修改后端逻辑
- 若问题本质是需求变更，先转 `requirements/CHANGE_PROCESS.md`

### 后端

- 只处理后端侧根因
- 修复接口、数据、权限、审计等后端问题
- 补后端回归测试

### 测试

- 提交可复现、有证据的 Issue
- 修复后基于分支 / PR 回归
- 回归通过后再推动关闭

## 7. 分支和 PR 清理规则

- 一个 Issue 一个分支
- PR 合并后删除远端分支
- 本地分支在合并后清理
- 不在已开 PR 的分支上继续叠加无关改动

建议：

- 仓库开启 GitHub 的 `Automatically delete head branches`

## 8. 最容易出问题的错误做法

以下做法不规范：

- 收到 Issue 后不分类，直接改代码
- 需求变更当作 Bug 直接修
- 直接在 `main` 上修复
- 修完不补测试
- 修完就直接关闭 Issue，不经过测试回归
- 一个修复分支上叠加多个不相关 Issue

## 9. 执行原则

一句话总结：

Issue 处理要遵循“读取 -> 分类 -> 分支修复 -> PR -> 回写 -> 转测试 -> 回归 -> 关闭”的完整链路。

关闭 Issue 不是开发修完的时点，而是测试回归通过后的时点。
